# 12-03 — Security Audit & OWASP Compliance

## Goal

Comprehensive security audit addressing OWASP Top 10, API authentication/authorization, data protection, and Swiss regulatory requirements. No critical vulnerabilities.

## Success Criteria (Non-Negotiable)

1. CORS headers audit: Verify `Access-Control-Allow-Origin` restricted to frontend origin (not `*`), `X-API-Key` required for all /api/* routes, OPTIONS preflight handled correctly
2. SQL injection prevention: Verify Prisma parameterized queries used everywhere; no raw SQL; no user input concatenated into queries
3. Authentication audit: API key validation working; constant-time comparison used to prevent timing attacks; no API keys exposed in logs
4. Rate limiting: Express middleware configured to limit requests per IP (e.g., 100 requests/15 minutes); /api/incidents POST limited (10 requests/5 minutes per IP)
5. Sensitive data audit: No PII (names, IPs, emails) logged; error messages don't expose system internals; no database credentials in frontend code
6. Input validation: All endpoints validate with Zod schemas; no incomplete validation; edge cases tested (empty strings, oversized payloads, null values)
7. Error handling: No stack traces exposed in production (only in development); generic error messages to clients; detailed logs server-side
8. OWASP Top 10 checklist completed and signed off: A01 (injection), A02 (broken auth), A03 (exposure), A04 (XML/XXE), A05 (broken access control), A06 (auth bypass), A07 (XSS), A08 (deserialization), A09 (logging), A10 (SSRF)
9. Security audit report committed to `docs/SECURITY_AUDIT.md` with findings, recommendations, and remediation steps

## Dependencies

- None (can run in parallel with 12-01 and 12-02)

## Critical Path

### 1. CORS Audit

- Check `src/utils/cors.ts`: Verify `CORS_ORIGIN` env var used in production, not hardcoded
- Test with curl: `curl -H "Origin: http://evil.com" http://localhost:3000/api/incidents` should fail
- Verify `X-API-Key` in `Access-Control-Allow-Headers`
- Test OPTIONS preflight: `curl -X OPTIONS http://localhost:3000/api/incidents -H "Origin: http://localhost:3000"` should return 200
- Verify correct headers returned

### 2. Authentication Audit

- Review `src/utils/auth.ts`: Verify `timingSafeEqual()` used for API key comparison
- Verify `X-API-Key` header required for all /api routes
- Test without API key: `curl http://localhost:3000/api/incidents` should return 401
- Verify no API key logged in error responses

### 3. Rate Limiting

- Install `express-rate-limit` (already in devDependencies)
- Add rate limiting middleware to `/api/incidents` POST route (max 10 requests/5 minutes per IP)
- Add general rate limiting to all /api routes (100 requests/15 minutes per IP)
- Test with load script: verify requests blocked when threshold exceeded, returns 429 status code

### 4. SQL Injection Prevention

- Review `src/api/services/incident.service.ts`: Verify all queries use Prisma (no raw SQL)
- Verify filter parameters (type, severity) properly enum-validated in schema
- Test with malicious input: `GET /api/incidents?type='; DROP TABLE incidents;--` should be rejected by Zod schema

### 5. Sensitive Data Audit

- Review error handling: Verify stack traces not exposed in production (`src/middleware/errorHandler.ts` checks NODE_ENV)
- Check logging: No PII in logs, no API keys, no passwords
- Review API responses: No sensitive fields returned unnecessarily
- Check environment variable usage: No secrets hardcoded (verify .env.example doesn't contain real values)

### 6. Input Validation Audit

- Test all schemas with edge cases: empty strings, null values, oversized payloads (>10MB rejected at Express middleware)
- Verify `description` field: min 10 chars enforced
- Verify `incident_type` and `severity`: enum values only
- Test pagination: `?limit=99999` should be capped at 100, negative page numbers rejected

### 7. Error Handling Consistency

- Verify all errors return structured JSON with `error` field
- Verify no raw Error.toString() output
- Test with invalid incident ID: `GET /api/incidents/invalid-uuid` should return 404 with message "Incident not found"

## Security Audit Implementation Details

### CORS Configuration Example
```typescript
// src/utils/cors.ts
const CORS_ORIGIN = process.env.CORS_ORIGIN || 'http://localhost:3000';

export function getCorsHeaders(origin?: string) {
  if (origin === CORS_ORIGIN) {
    return {
      'Access-Control-Allow-Origin': origin,
      'Access-Control-Allow-Methods': 'GET,POST,PATCH,DELETE,OPTIONS',
      'Access-Control-Allow-Headers': 'Content-Type,X-API-Key',
      'Access-Control-Max-Age': '86400',
    };
  }
  return {};
}
```

### Authentication Example
```typescript
// src/utils/auth.ts
import { timingSafeEqual } from 'crypto';

export function validateApiKey(key?: string): boolean {
  const expected = process.env.API_KEY;
  if (!key || !expected) return false;
  
  try {
    return timingSafeEqual(Buffer.from(key), Buffer.from(expected));
  } catch {
    return false;
  }
}
```

### Rate Limiting Example
```typescript
// src/api/index.ts
import rateLimit from 'express-rate-limit';

const apiLimiter = rateLimit({
  windowMs: 15 * 60 * 1000, // 15 minutes
  max: 100, // limit each IP to 100 requests per windowMs
  message: 'Too many requests, please try again later',
});

const postLimiter = rateLimit({
  windowMs: 5 * 60 * 1000, // 5 minutes
  max: 10, // limit each IP to 10 POST requests
});

app.use('/api/', apiLimiter);
app.post('/api/incidents', postLimiter, createIncident);
```

## Risks

- **Rate limiting might block legitimate load test traffic**; configure allowlist for test IPs
- **CORS misconfiguration could break frontend**; test with actual frontend origin before deployment
- **Over-strict rate limiting could impact API usability**; balance security with DX
