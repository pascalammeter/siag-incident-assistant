# 12-01 — Integration & Unit Tests

## Goal

Achieve >80% code coverage on new backend code (API service layer, schemas, utilities) with comprehensive unit and integration tests covering all CRUD flows and error scenarios.

## Success Criteria (Non-Negotiable)

1. Unit tests written for `IncidentService` (all 6 methods), `Zod` schemas, and utility functions (`cors.ts`, `auth.ts`, `fileDownload.ts`)
2. Integration tests cover end-to-end flows: Create → Save → List → Retrieve → Delete
3. Integration tests cover Wizard API submission: Wizard form → POST /api/incidents → Database persisted → GET returns correct data
4. Coverage report shows >80% coverage for `src/api/**` (services, schemas, middleware)
5. All new tests use Vitest + Supertest with proper mocking strategy (Prisma mocked, real schema validation)
6. 106 Phase 11 tests continue passing; 180+ total tests by end of plan
7. No console warnings or TypeScript errors; tests runnable via `npm test` with clear output

## Dependencies

- None (can run in parallel with 12-02 and 12-03)

## Critical Path

1. Add coverage reporting to `vitest.config.ts` (Vitest built-in coverage with v8 provider)
2. Write unit tests for `IncidentService`: createIncident, getIncidentById, updateIncident, deleteIncident, listIncidents, exportToPDF (6 test files)
3. Write unit tests for `Zod` schemas: CreateIncidentInputSchema, UpdateIncidentInputSchema, ListIncidentsQuerySchema (1 test file)
4. Write unit tests for utility functions: `getCorsHeaders()`, `validateApiKey()`, `fileDownload()` (1 test file)
5. Write integration tests for API endpoints: POST /api/incidents, GET /api/incidents, GET /api/incidents/:id, PATCH /api/incidents/:id, DELETE /api/incidents/:id (1 integration test file)
6. Write integration test for full wizard flow: Frontend form submission → API → Database → Retrieve (1 e2e integration test file)
7. Run `npm test:coverage` and verify >80% coverage; document coverage report
8. All tests passing, committed to `tests/` directory with clear naming convention

## Test Structure

- **`tests/api/incidents.service.test.ts`** — IncidentService unit tests (30 tests, 5-8 minutes runtime)
  - Test createIncident: valid payload, invalid type, missing required fields
  - Test getIncidentById: found, not found, invalid UUID
  - Test updateIncident: partial update, full update, concurrent updates
  - Test deleteIncident: successful, not found, already deleted
  - Test listIncidents: pagination, filtering by type/severity, sorting
  - Test exportToPDF: valid incident, PDF generation, error handling

- **`tests/schemas/incident.schema.test.ts`** — Zod schema validation (20 tests, 2-3 minutes)
  - Test CreateIncidentInputSchema: valid inputs, invalid enums, missing fields, string length constraints
  - Test UpdateIncidentInputSchema: partial updates, invalid fields, type coercion
  - Test ListIncidentsQuerySchema: valid pagination, boundary values, invalid values

- **`tests/utils/auth.cors.fileDownload.test.ts`** — Utility functions (15 tests, 2-3 minutes)
  - Test getCorsHeaders: correct headers, allowlist enforcement, missing origin header
  - Test validateApiKey: valid key, invalid key, timing-safe comparison
  - Test fileDownload: filename encoding, correct headers, stream errors

- **`tests/api/integration.e2e.test.ts`** — Full CRUD + export integration (25 tests, 10-15 minutes)
  - Test POST /api/incidents: create and retrieve, validate response structure
  - Test GET /api/incidents: list all, filter by type, paginate results
  - Test GET /api/incidents/:id: found, not found, soft-deleted items
  - Test PATCH /api/incidents/:id: update severity, update playbook, partial updates
  - Test DELETE /api/incidents/:id: soft delete, verify deletedAt timestamp
  - Test export endpoints: JSON export validity, PDF generation, download headers

- **`tests/hooks/useIncident.integration.test.ts`** — Wizard → API flow (15 tests, 5-8 minutes)
  - Test useIncident hook: create new incident, resume existing, type selector
  - Test wizard form submission: payload validation, error handling
  - Test localStorage fallback: API unavailable scenario, graceful degradation
  - Test data persistence: saved to database, retrieved correctly

## Test Implementation Details

### Mocking Strategy
- Mock Prisma client with `vi.mock('@prisma/client')`
- Real Zod schema validation (no mocking)
- Mock Express Request/Response with Supertest
- Use test database or in-memory fixtures for integration tests

### Coverage Goals
- **IncidentService**: 100% (all methods, all branches)
- **Schemas**: 100% (valid + invalid cases)
- **Middleware**: 80%+ (error handling, validation)
- **Routes**: 75%+ (happy paths + error cases)
- **Overall**: >80% for `src/api/**`

### Performance Considerations
- Run tests in parallel (Vitest default)
- Set `testTimeout: 30000` for database-dependent tests
- Use test fixtures to avoid database setup/teardown overhead
- Profile test runtime; target <3 minutes for full suite

## Risks

- **Prisma mocking complexity**: Ensure `vi.mock()` calls intercept correctly; test with real Prisma client if mocking fails
- **Database connection timeout in CI**: Use `testTimeout: 30000` (already in vitest.config.ts)
- **Coverage thresholds difficult to meet**: Focus on IncidentService and schemas first; utility functions are simple

## Assumptions

- Vitest coverage provider (v8) available in node environment
- Supertest can mock Express app without running server
- Prisma mocking with `vi.mock()` works for all methods used (create, findFirst, findMany, update, count)
