# 12-02 — Load Testing & Performance Optimization

## Goal

Verify API can handle production traffic (100+ concurrent requests, sustained load over 5 minutes) with response times meeting SLA (avg <200ms, p95 <500ms). Identify and fix bottlenecks.

## Success Criteria (Non-Negotiable)

1. k6 load test script written for 3 scenarios: read-heavy (100 concurrent GET), write-heavy (50 concurrent POST over 5min), mixed (70/30 read/write for 10min)
2. Load test results: 100 concurrent requests to GET /api/incidents: avg <200ms, p95 <500ms, p99 <1000ms, 0 errors
3. Load test results: 50 concurrent POST creates over 5 minutes: all complete successfully, avg response time <500ms, 0 timeouts
4. Load test results: 10-minute sustained load test: no memory leaks (heap grows <10% over test duration), no database connection pool exhaustion
5. Database query performance audit: Identify any N+1 queries or missing indexes; document findings
6. Optimization recommendations documented: if response times exceed SLA, provide specific fixes (caching, indexes, query optimization)
7. Load test script committed to `tests/load/` with clear documentation on how to run and interpret results

## Dependencies

- 12-01 (tests must pass first; baseline for performance)

## Critical Path

1. Install k6 CLI locally (`brew install k6` or download from k6.io)
2. Write k6 script for read-heavy test: 100 concurrent users for 30 seconds, ramping up over 10 seconds, hitting GET /api/incidents?limit=50
3. Write k6 script for write-heavy test: 50 concurrent users creating incidents (POST /api/incidents) over 5 minutes with ramp-up
4. Write k6 script for memory leak detection: sustained mixed load for 10 minutes, capture heap metrics every 10 seconds
5. Run tests locally against Express server running on `localhost:3000`
6. Capture results: response times (avg, p50, p95, p99), error rate, throughput, connection pool status
7. If response times exceed SLA: analyze with Prisma logging enabled, check for N+1 queries, add database indexes if needed
8. If memory leak detected: check for unclosed database connections or event listener leaks
9. Document findings in `tests/load/LOAD_TEST_RESULTS.md` with graphs and recommendations
10. Commit k6 scripts to `tests/load/` with instructions for CI/CD integration

## Load Test Scenarios

### Scenario A: Read-Heavy (GET /api/incidents)
- 100 concurrent users, ramp-up 10 seconds, duration 30 seconds
- Query params vary: ?limit=10, ?limit=50, ?type=ransomware, ?severity=critical
- **Target:** avg <200ms, p95 <500ms, error rate 0%

### Scenario B: Write-Heavy (POST /api/incidents)
- 50 concurrent users, ramp-up 5 seconds, duration 5 minutes
- Each user creates 1-3 incidents per test run
- Payload varies incident type and severity
- **Target:** avg <500ms, p95 <1000ms, error rate 0%, all completes

### Scenario C: Sustained Load (Mixed Read/Write)
- 70 users reading, 30 users writing, ramp-up 15 seconds, duration 10 minutes
- Monitors heap size, database connections, memory stability
- **Target:** heap growth <10%, connection pool stable, no leaks

### Performance Metrics to Capture
- **Response times:** avg, p50, p95, p99, max
- **Throughput:** requests/second, duration
- **Errors:** count, rate, types
- **Memory:** initial heap, peak heap, final heap (for leak detection)
- **Database connections:** pool size, wait time
- **Custom metrics:** API-specific (e.g., incident creation latency)

## Performance Optimization Strategy

### If response times exceed SLA:

1. **Enable Prisma logging** to identify slow queries
   ```bash
   DATABASE_URL="postgresql://..." DATABASE_DEBUG=* npm run dev:backend
   ```

2. **Check for N+1 queries**
   - Review IncidentService methods for unnecessary queries
   - Use Prisma `include` / `select` to batch queries
   - Example: `listIncidents` should use single query, not loop

3. **Add missing indexes** to `prisma/schema.prisma`
   ```prisma
   @@index([incident_type])
   @@index([severity])
   @@index([createdAt])
   ```

4. **Check database connection pooling**
   - Verify Prisma adapter configured for Neon serverless
   - Monitor connection pool exhaustion in k6 results

5. **Optimize Puppeteer PDF generation** (if export is slow)
   - Reduce PDF quality if needed
   - Cache static resources (logos, fonts)
   - Consider async generation + background job queue

## Risks

- **k6 requires Go runtime locally**; Docker alternative available if Go not installed
- **Database can become bottleneck**; depends on Neon PostgreSQL performance
- **Network latency to Neon can skew results**; measure latency separately

## Assumptions

- Express server can run locally on localhost:3000 without issues
- Neon PostgreSQL connection pooling configured (Prisma adapter already includes pooling)
- k6 available in test environment (can use k6 cloud alternative if needed)
