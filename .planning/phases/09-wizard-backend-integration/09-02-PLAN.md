# 09-02 PLAN: Data Migration Layer

**Objective:** Build an automatic data migration system that safely transfers v1.0 localStorage incidents to the v1.1 backend API on first app load. Users with v1.0 incidents see them instantly available in the API, with silent success notification.

**Purpose:** Enable seamless upgrade path from v1.0 MVP to v1.1 backend. Existing incidents are preserved and migrated without user intervention.

**Output:** Migration hook and utility functions ready for deployment with v1.1. All v1.0 incidents are accessible via API after first load.

## Context

### v1.0 ã v1.1 Schema Mapping

**v1.0 localStorage structure** (from Phase 5):
```json
{
  "currentStep": number,
  "noGoConfirmed": boolean,
  "einstieg": {...},
  "erfassen": {
    "erkennungszeitpunkt": "ISO8601",
    "erkannt_durch": "string",
    "betroffene_systeme": ["string"],
    "erste_auffaelligkeiten": "string",
    "loesegeld_meldung": boolean
  },
  "klassifikation": {
    "q1SystemeBetroffen": "ja|nein",
    "q2PdBetroffen": "ja|nein",
    "q3AngreiferAktiv": "ja|nein|unbekannt",
    "incidentType": "ransomware|phishing|...",
    "severity": "KRITISCH|HOCH|MITTEL"
  },
  "reaktion": {
    "completedSteps": ["string"]
  },
  "kommunikation": {...},
  "dokumentation": {...}
}
```

**v1.1 API schema** (from Prisma):
```json
{
  "id": "uuid",
  "createdAt": "ISO8601",
  "updatedAt": "ISO8601",
  "incident_type": "ransomware|phishing|ddos|data_loss|other",
  "severity": "critical|high|medium|low",
  "erkennungszeitpunkt": "ISO8601?",
  "erkannt_durch": "string?",
  "betroffene_systeme": ["string"]?,
  "erste_erkenntnisse": "string?",
  "q1": 0|1,
  "q2": 0|1,
  "q3": 0|1,
  "playbook": {...},
  "regulatorische_meldungen": {...},
  "metadata": {...}
}
```

**Mapping logic:**
- `erfassen.erkennungszeitpunkt` ã `erkennungszeitpunkt`
- `erfassen.erkannt_durch` ã `erkannt_durch`
- `erfassen.betroffene_systeme` ã `betroffene_systeme`
- `erfassen.erste_auffaelligkeiten` ã `erste_erkenntnisse`
- `klassifikation.incidentType` ã `incident_type` (enum rename: 'datenverlust' ã 'data_loss')
- `klassifikation.severity` ã `severity` (enum rename: 'KRITISCH' ã 'critical', 'HOCH' ã 'high', 'MITTEL' ã 'medium')
- `klassifikation.q1SystemeBetroffen` ã `q1` (convert 'ja'/'nein' ã 1/0)
- `klassifikation.q2PdBetroffen` ã `q2`
- `klassifikation.q3AngreiferAktiv` ã `q3` (convert 'ja'/'nein'/'unbekannt' ã 1/0/null)
- `reaktion.completedSteps` ã `playbook.checkedSteps` (preserve array)
- Entire wizard state ã `metadata.fullState` (backup of original v1.0 state)

### Migration Flow

1. App loads (RootLayout mounts)
2. useMigration() hook checks for v1.0 `siag-wizard-state` in localStorage
3. If found:
   - Parse v1.0 state
   - For each "incident" (could be singular current state):
     - Transform fields using mapping logic
     - POST to `/api/incidents`
     - If 201: mark for deletion
   - After all POSTs succeed: delete `siag-wizard-state` from localStorage
   - If any POST fails with 5xx or network error:
     - Keep `siag-wizard-state` in localStorage
     - Set retry flag for next load
     - Show warning toast
4. If any POST fails with 4xx (validation):
   - Show error toast with details
   - Keep data in localStorage for manual fix
5. Success: show toast "X incidents migrated to API"
6. Migration runs async, UI not blocked

### Retry Logic

If API unavailable during migration:
- v1.0 data stays in localStorage
- New flag: `siag-migration-pending` set to true
- On next app load: retry migration
- User never loses data

  <action>
Create `useMigration()` hook that orchestrates the migration on app load.

Hook interface:
```typescript
export function useMigration() {
  // Hook logic
  // Returns nothing (runs as side effect)
}
```

Hook behavior:

1. **On mount (useEffect):**
   - Check if migration already completed: `localStorage.getItem('siag-migration-completed') === 'true'`
   - If yes, return (skip migration)
   - If no, continue

2. **Load and validate v1.0 state:**
   - Read `siag-wizard-state` from localStorage
   - If not found, set flag `siag-migration-completed=true`, return
   - Parse JSON; if error, skip (corrupted data)

3. **Transform incidents:**
   - Call `migrateIncidents(v1State)` from migration.ts
   - Get array of `CreateIncidentInput`

4. **Migrate to API (async):**
   - For each incident:
     - Call `useIncident().createIncident(input)`
     - Wait for 201 response
     - Track count of successful migrations
   - If any fail with 5xx or network error:
     - Set `siag-migration-pending=true` in localStorage
     - Show error toast "Migration failed, will retry on next load"
     - Return (don't delete v1.0 data)
   - If any fail with 4xx (validation):
     - Show error toast with field details
     - Keep data in localStorage for manual review
     - Continue to next incident (don't block entire migration)

5. **Cleanup (on success):**
   - Delete `siag-wizard-state` from localStorage
   - Set `siag-migration-completed=true`
   - Show success toast: "X incidents migrated to API" (only if X > 0)

6. **Error handling:**
   - Catch all errors and log (don't throw, migration must not break app)
   - Show toast on error (via sonner library, assumed available from Phase 1)
   - Migration runs async in background

7. **Dependencies:**
   - Use `useIncident()` hook from Phase 9 Plan 1
   - Use migration.ts transformation functions
   - Use toast from UI library
   - Run in useEffect with empty deps (mount once on app load)

Type safety: All incident inputs are `CreateIncidentInput` type, validated before API call.

## Threat Model

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|-----------|
| T-09-05 | D (Denial of Service) | Migration loop | mitigate | Retry flag prevents infinite loops; migration marked completed after success. If stuck, user can manually delete `siag-wizard-state` from localStorage. |
| T-09-06 | A (Authentication) | API calls | accept | No API key yet (Phase 12); migration is one-time on load, assumes trusted client context (user's own browser). |
| T-09-07 | I (Information Disclosure) | v1.0 data in localStorage | mitigate | v1.0 data deleted after migration; temporary window where both old and new data exist, but both are user-controlled localStorage (no secrets). |
| T-09-08 | T (Tampering) | Corrupted v1.0 state | mitigate | JSON parse errors caught and logged; corrupted entries skipped; app continues to function. Manual review flag guides users to check results. |
