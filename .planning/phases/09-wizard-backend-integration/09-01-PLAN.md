# 09-01 PLAN: useIncident() Hook Implementation

**Objective:** Create a custom React hook that provides API-backed incident state management with graceful localStorage fallback. This hook replaces direct localStorage usage from v1.0 and becomes the primary storage layer for all incident operations (create, read, update, delete).

**Purpose:** Enable the wizard and incident list to persist data to the backend API while maintaining the ability to function offline or when the API is temporarily unavailable.

**Output:** Fully typed, tested `useIncident()` hook ready for integration with wizard components and incident list page.

## Context

### Canonical References

- **Prisma Schema:** `./prisma/schema.prisma` — Incident model defines all fields
- **Phase 8 API Routes:** `/server/api/incidents.ts` (POST, GET/:id, PATCH/:id, DELETE/:id)
- **v1.0 Wizard Hook:** `src/components/wizard/WizardContext.tsx` — localStorage pattern to replace
- **Existing Types:** `src/lib/wizard-types.ts` — will extend with incident-specific types

### HTTP Client Pattern

All API calls go through a centralized client that:
1. Handles network errors gracefully (timeout, connection refused, etc.)
2. Converts HTTP errors to JS exceptions with `.status` property
3. Falls back to localStorage if API unavailable
4. Returns properly typed responses

### Fallback Strategy

localStorage is used when:
- Network error occurs (fetch throws or times out)
- API returns 5xx status code (server error)
- API returns 401/403 (auth error, deferred to Phase 12)

localStorage is NOT used when:
- API returns 4xx client error (validation error — bubble to UI)
- Explicit offline mode flag set

<task type="auto">
  <name>Task 4: Implement useIncident() Hook with Fallback Logic</name>
  <files>src/hooks/useIncident.ts</files>
  <action>
Create the main `useIncident()` hook that provides incident state management with API-first strategy and localStorage fallback.

Hook should return an object with:

```typescript
{
  incident: Incident | null        // current incident (from API or localStorage)
  isLoading: boolean               // true while API request in progress
  error: string | null             // error message if operation failed
  
  // CRUD operations
  createIncident(input: CreateIncidentInput): Promise<Incident>
  getIncident(id: string): Promise<Incident>
  updateIncident(id: string, input: UpdateIncidentInput): Promise<Incident>
  deleteIncident(id: string): Promise<void>
  listIncidents(filters?: ListIncidentsFilters): Promise<Incident[]>
  
  // Fallback detection
  isOffline: boolean               // true if last operation used localStorage fallback
  clearError(): void               // manual error dismissal
}
```

**Implementation strategy:**

1. **Primary flow (API available):**
   - Call IncidentAPI method
   - Set isLoading=true, wait for response
   - If 2xx: update incident state, set error=null, isOffline=false
   - Return resolved incident

2. **Fallback flow (API unavailable):**
   - Catch APIError or network error
   - If 5xx or network error: fall back to localStorage
     - Write incident to localStorage key: `siag-incident-{id}`
     - Set isOffline=true
     - Show warning toast (via caller)
   - If 4xx: bubble error to UI (validation error, not offline)
   - Set error message for UI

3. **localStorage structure:**
   - Key: `siag-incident-{id}` for individual incidents
   - Key: `siag-incidents` for list (array of incident objects)
   - Key: `siag-incident-fallback` (flag indicating fallback was used)

4. **Type safety:**
   - Use incident-types.ts for all inputs/outputs
   - Generic state: `incident: Incident | null`
   - Error state includes `.message` for display

5. **Edge cases:**
   - Closing hook (cleanup): ensure pending requests don't update unmounted state (AbortController)
   - Concurrent operations: serialize updates (use prev state from setState callback)
   - Corrupted localStorage: catch JSON.parse errors, ignore bad data

Per USER MEMORY.md "Critical rule": Do NOT swap out `useWizard()` yet. Instead, create `useIncident()` as a parallel hook. Phase 9 integration task (separate plan) will wire it into wizard.

Write the hook using `useState` + `useCallback` for methods. Do NOT use useReducer yet (simpler for Phase 9).

| T-09-04 | D (Denial of Service) | localStorage quota | accept | localStorage ~5MB per domain; Phase 9 stores 1-100 incidents (~10-500KB), no DoS risk. Future: implement quota warning if needed. |
