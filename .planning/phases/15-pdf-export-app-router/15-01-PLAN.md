<task type="auto">
  <name>Task 2: Design Route Handler Logic</name>
  <files>src/app/api/incidents/[id]/export/pdf/route.ts (design only)</files>
  <action>
    Design (do not implement yet) the route handler structure. It should:
    
    1. **Handler signature:** Use Next.js 15 App Router pattern
       - `export async function GET(request: NextRequest, { params }: { params: Promise<{ id: string }> })`
       - Await params to get the incident ID (Next.js 15+ pattern)
    
    2. **API key validation:** Call `validateApiKey(request)` from `src/app/api/_helpers.ts` (same as other incident routes)
       - If auth fails, return early
    
    3. **Fetch incident:** Call `IncidentService.getIncidentById(id)`
       - If not found, return `errorResponse('Incident not found', 404)`
       - If found, proceed to PDF generation
    
    4. **Generate PDF:**
       a. Get browser instance: `const browser = await getBrowserInstance()`
       b. Create a new page: `const page = await browser.newPage()`
       c. Combine HTML: concatenate title page + details page HTML
       d. Set content: `await page.setContent(combinedHTML, { waitUntil: 'networkidle0' })`
       e. Generate PDF: `const pdfBuffer = await page.pdf({ format: 'A4', margin: { top: 20, right: 20, bottom: 20, left: 20 } })`
       f. Close page: `await page.close()`
    
    5. **Response headers:**
       - `Content-Type: application/pdf`
       - `Content-Disposition: attachment; filename="incident-${id}-${date}.pdf"`
       - Return buffer as response body
    
    6. **Error handling:**
       - Wrap PDF generation in try/catch
       - Log errors to console with context
       - Return `errorResponse('Failed to generate PDF', 500)` on failure
       - Do NOT leave browser page open (always close, even on error)
    
    7. **Return type:** Return a Response object with binary data (use `new Response(pdfBuffer, { headers: {...} })`)
  </action>
  <verify>Design document outlines all 7 sections above with correct Next.js App Router patterns</verify>

       ```
       import { getBrowserInstance } from '@/lib/puppeteer-singleton'

       import { IncidentService } from '@/api/services/incident.service'

       ```
    
    3. **OPTIONS handler:** Include CORS preflight (copy pattern from `/api/incidents/[id]/route.ts`)
       - Call `handleOptions()` from `_helpers.ts`
    
    4. **GET handler implementation:**
       - Extract id from params: `const { id } = await params`
       - Validate API key: `const authError = validateApiKey(request); if (authError) return authError;`
       - Fetch incident: `const incident = await IncidentService.getIncidentById(id)`
       - Check if exists: `if (!incident) return errorResponse('Incident not found', 404)`
       - Wrap PDF generation in try/catch:
         * Get browser: `const browser = await getBrowserInstance()`
         * Create page: `const page = await browser.newPage()`
         * Build HTML: combine `generateTitlePageHTML(incident)` + `generateIncidentDetailsHTML(incident)`
         * Set page content: `await page.setContent(combinedHTML, { waitUntil: 'networkidle0' })`
         * Generate PDF: `const pdfBuffer = await page.pdf({ format: 'A4', margin: { top: 20, right: 20, bottom: 20, left: 20 } })`
         * Close page: `await page.close()`
       - Build response headers:
         * `Content-Type: application/pdf`
         * `Content-Disposition: attachment; filename="incident-${id}-${new Date(incident.createdAt).toISOString().split('T')[0]}.pdf"`
       - Return response: `new Response(pdfBuffer, { status: 200, headers: { 'Content-Type': 'application/pdf', 'Content-Disposition': contentDisposition } })`
       - Catch errors: Log to console, return `errorResponse('Failed to generate PDF', 500)`, always close page
    
    5. **Type safety:** Use TypeScript strict mode — annotate all parameters and return types
    
    6. **Code style:** Match existing codebase patterns (from `/api/incidents/[id]/route.ts`):
       - Use descriptive console.error logs with bracket prefix: `console.error('[GET /api/incidents/:id/export/pdf]...')`
       - Return early for validation/auth failures
       - Wrap async operations in try/catch
    
    **Gotcha:** Page.pdf() returns a Buffer, which is already binary data. When returning in Response, ensure the response has the correct headers so the browser downloads it as a file, not renders it inline.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API Route | Untrusted incident ID (path parameter) and API key (header) |
| API Route → Browser | Untrusted HTML content (could contain XSS if incident data not sanitized) |
| API Route → Database | Trusted — Prisma ORM with parameterized queries |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | **S**poofing | API Key header | Mitigate | Validate X-API-Key via validateApiKey() — same pattern used in other incident routes (see /api/incidents/[id]/route.ts) |
| T-15-02 | **T**ampering | Incident ID path parameter | Mitigate | Use Prisma parameterized queries in getIncidentById() — UUID validation happens at DB level |
| T-15-03 | **R**epudiation | PDF generation | Accept | No audit log required in v1.1; logging added in v1.2 as per CONTEXT.md deferred ideas |
| T-15-04 | **I**nformation Disclosure | PDF file content | Mitigate | PDF generated contains incident data already protected by API key; no additional PII encryption in v1.1 (Phase 14 schema handles field persistence) |
| T-15-05 | **D**enial of Service | Browser resource exhaustion | Mitigate | Puppeteer singleton limits concurrent browser instances (1 per Lambda container); large PDF generation is bounded by Puppeteer timeouts; consider adding request timeout at route level in v1.2 |
| T-15-06 | **E**levation of Privilege | PDF generation as system process | Mitigate | Puppeteer runs in sandbox mode by default; no privilege escalation possible from user-supplied incident data |

</threat_model>
