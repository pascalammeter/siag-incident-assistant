<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer → GitHub Secrets | Developers write code; secrets are protected by GitHub access control |
| GitHub Actions → Vercel | CI/CD uses VERCEL_TOKEN to deploy; token must be short-lived or scoped |
| GitHub Actions → Neon DB | CI/CD uses DATABASE_URL_CI; should be separate from production DB |
| Main branch → Production | Branch protection enforces: PR review + passing tests before deploy |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | **Spoofing** | GitHub Secrets | Mitigate | GitHub access control + 2FA enforcement on repo (admin setting) |
| T-20-02 | **Tampering** | .env.example | Accept | File is public template (no secrets); values in runtime are protected by GitHub Secrets |
| T-20-03 | **Repudiation** | Branch commits | Mitigate | Branch protection + PR reviews create audit trail of all code changes |
| T-20-04 | **Information Disclosure** | API_KEY in logs | Mitigate | API_KEY_CI is test-only (sk_test_ prefix); logs filtered via GitHub Actions masking (automatic) |
| T-20-05 | **Denial of Service** | Database credentials | Mitigate | DATABASE_URL_CI points to separate test DB; production DB (DIRECT_URL) never in CI logs |
| T-20-06 | **Elevation of Privilege** | Force pushes to main | Mitigate | Branch protection rule: `Include administrators` enabled — prevents all force pushes |

</threat_model>
