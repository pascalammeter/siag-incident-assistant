# Phase 14 Plan 01: Consolidate Zod Schemas

## Objective

Eliminate schema fragmentation by consolidating two conflicting Zod schema definitions into a single unified schema that supports both API input validation and wizard field persistence.

## Execution Context

**Current State:**
- Two competing schema files exist with overlapping but incomplete definitions
- `src/api/schemas/incident.schema.ts` (lines 20-27): API-focused schema with missing wizard fields
- `src/lib/schemas/incident.schema.ts` (lines 3-13): Wizard-focused schema with wizard-specific fields
- IncidentService imports only from `src/api/schemas/incident.schema.ts` (line 2)
- No schema for data_loss incident type in getPlaybook switch (src/lib/playbook-data.ts lines 191-204)

**Problem:**
- Wizard fields (erkennungszeitpunkt, erkannt_durch, erste_erkenntnisse, betroffene_systeme, q1/q2/q3) are dropped during API create because CreateIncidentInputSchema (lines 20-27) doesn't include them
- Database schema (prisma/schema.prisma lines 19-28) supports all fields but API validation rejects them
- This causes wizard data loss when incidents are saved
