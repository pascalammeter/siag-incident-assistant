## Playbook Structure Reference

From src/lib/playbook-data.ts:
- Playbook interface: { incidentType: string, phases: PlaybookPhase[] }
- PlaybookPhase: { id: string, title: string, steps: PlaybookStep[] }
- PlaybookStep: { id: string, text: string, role: Role, noGoWarning?: string }
- Total steps per playbook: exactly 25 (requirement M1.2, M1.3)
- 4 phases per playbook (Erkennung, Eindaemmung, Untersuchung, Kommunikation pattern)

## DDoS Playbook Content Strategy

Phase 1: Erkennung (Detection) — ~6 steps
- Traffic anomaly detection (bandwidth spike, packet rate increase)
- Source analysis (botnet IPs, geographic origin, ASNs)
- Attack vector identification (volumetric, protocol, application-layer)
- Service impact assessment (which services unavailable, customer impact)
- Upstream provider notification initiation (alert ISP/CDN before mitigation)
- Incident timeline (when detected, duration so far)

Phase 2: Eindaemmung (Mitigation) — ~7 steps
- Activate DDoS mitigation service (if subscribed): Cloudflare, AWS Shield, etc.
- ISP null-routing (ask ISP to drop attack traffic at upstream)
- Rate-limiting on affected services (per-IP limits, challenge-response)
- CDN activation (if not already active, shift DNS to CDN to absorb traffic)
- Geo-blocking (if attack comes from specific countries, restrict access)
- Upstream provider mitigation (coordinate with ISP for BGP blackhole)
- Communication with customers (status page, ETA for resolution)

Phase 3: Untersuchung (Investigation) — ~6 steps
- Attack pattern analysis (frequency, duration, payload, evolution)
- Attacker identification (botnet signature, previous attacks, known group)
- Forensic log preservation (NetFlow, DNS logs, web server logs)
- Business impact assessment (revenue loss, SLA breach, customer complaints)
- Incident duration tracking (total downtime, recovered services)
- Root cause analysis (how did attacker obtain botnet, vulnerability exploited)

Phase 4: Kommunikation (Communication) — ~6 steps
- Status page update (real-time incident status for customers)
- Customer notification (if SLA breached, compensation/credits)
- Internal communication (incident report to leadership)
- Regulatory notification (if financial/critical infrastructure, report to authorities)
- Post-incident review (prevent future attacks, infrastructure improvement)
- Lessons learned documentation (what worked, what didn't)

## Data Loss Playbook Content Strategy

Phase 1: Erkennung (Detection) — ~6 steps
- Data exfiltration indicators (large file transfers, anomalous downloads, encryption)
- Backup system check (verify backups are intact and not encrypted)
- Database integrity check (test restores, verify no data corruption)
- User report analysis (if data loss detected via user action)
- Data classification (what data was lost: PII, financial, IP, operational)
- Scope assessment (how much data, how many records, which systems)

Phase 2: Eindaemmung (Containment) — ~7 steps
- Revoke attacker access (disable accounts, API keys, service principals)
- Isolate affected systems (remove from network, prevent further exfiltration)
- Patch critical vulnerabilities (if attack vector was known vuln, patch immediately)
- Re-protect backups (move offline, verify integrity, test restoration)
- Access control hardening (remove unnecessary access, enable MFA)
- Monitor for re-infection (watch for signs attacker still has access)
- Preserve forensic evidence (disk snapshots, logs, memory dumps)

Phase 3: Untersuchung (Investigation) — ~6 steps
- Timeline reconstruction (when was data accessed, how long had attacker access, what was exfiltrated)
- Attack vector analysis (how did attacker get in: phishing, vuln, insider, misconfiguration)
- Data scope confirmation (which data tables/files were accessed, partial or complete)
- Affected parties identification (if PII, which individuals, how many records)
- Attacker identification (external threat group, insider, accident)
- Incident cost estimation (breach notification costs, remediation, legal)

Phase 4: Kommunikation (Communication) — ~6 steps
- Regulatory notification (GDPR, DSG, CCPA if PII exposed; ISG 24h for ransomware component)
- Affected user notification (timeline per regulation: GDPR 72h, DSG 30d)
- Law enforcement reporting (FBI if applicable, local police)
- Internal communication (leadership, board, PR)
- Credit monitoring notification (if PII with payment cards)
- Post-incident review (security improvements, backup strategy review)

All content must be:
- Actionable and specific to incident type
- Role-assigned (IT-Leiter, CISO, CEO, Forensik)
- Aligned with Swiss regulatory requirements (ISG, DSG, FINMA)
- Non-contradictory within playbook
