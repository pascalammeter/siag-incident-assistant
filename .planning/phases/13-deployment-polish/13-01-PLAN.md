# Phase 13-01 Plan: Vercel + Neon Production Deployment

## Overview
Deploy v1.1 backend as Vercel Functions, provision Neon PostgreSQL with connection pooling configured for serverless, run migrations automatically on first deploy, and enable backups. Configure environment variables for production (database URL, API keys, CORS origins).

## Success Criteria (MUST be TRUE)
1. ✅ Backend deployed as Vercel Functions; accessible at production Vercel URL
2. ✅ Neon PostgreSQL provisioned with database created; connection pooling configured for serverless environments
3. ✅ Prisma migrations run successfully on first production deploy without manual intervention
4. ✅ Environment variables configured in Vercel for DATABASE_URL, JWT_SECRET, API_KEY, CORS_ORIGINS
5. ✅ Backups enabled on Neon database; daily snapshots retained for 7 days
6. ✅ Frontend updated with production API URL (remove localhost:3000, point to Vercel production)

## Tasks

### 1. Neon Database Setup (2 hours)
- [ ] Create Neon project if not exists; note project ID
- [ ] Provision new PostgreSQL database (v16+) in primary region
- [ ] Configure connection pooling for Prisma (min 2, max 10 connections)
- [ ] Generate connection string with pooling parameters
- [ ] Enable daily backups (7-day retention)
- [ ] Store connection string securely (will be added to Vercel env vars)

### 2. Prisma & Migrations Configuration (2 hours)
- [ ] Update `DATABASE_URL` in `.env.production` with Neon connection string
- [ ] Verify prisma schema matches Phase 12 (incidents, incidents_types, audit logs tables)
- [ ] Run `npm run prisma:migrate` in local dev environment to validate migrations
- [ ] Add `DATABASE_URL` to Vercel environment variables
- [ ] Configure Vercel build step: `npm run prisma:migrate` runs before deployment
- [ ] Test deploy with fresh database using `vercel preview`

### 3. Environment Variables Configuration (1.5 hours)
- [ ] Create production environment in Vercel project settings
- [ ] Add required secrets:
  - `DATABASE_URL` — Neon connection string (with pooling)
  - `JWT_SECRET` — 32-character random string for signing tokens
  - `API_KEY` — Admin API key for internal requests
  - `CORS_ORIGINS` — Production domain(s): "https://siag-incident-assistant.vercel.app"
  - `NODE_ENV` — "production"
- [ ] Verify all vars are marked as secret (not exposed to client)
- [ ] Document required environment variables in `.env.example`

### 4. Vercel Deployment Configuration (1.5 hours)
- [ ] Ensure `vercel.json` (or `vercel.ts`) correctly configured:
  - Build command: `npm run build`
  - Output directory: `.next` or `.vercel/output` (depending on config)
  - Functions runtime: Node.js 20 LTS
  - Regions: Default (auto-select based on closest to Neon)
- [ ] Configure function memory: 1024 MB for API routes
- [ ] Set function timeout: 60 seconds (default; increase to 120s if needed for long-running exports)
- [ ] Enable request body size limit: 10 MB (for PDF exports)

### 5. Frontend API URL Update (1 hour)
- [ ] Update `NEXT_PUBLIC_API_URL` environment variable in Vercel to production API URL
- [ ] If using `.env.local`, update with production Vercel Functions URL
- [ ] Test that wizard → backend communication works (create incident, verify in database)
- [ ] Verify CORS headers are correct (no 403 errors on API calls)

### 6. Deployment & Smoke Test (1.5 hours)
- [ ] Deploy to Vercel production using `vercel deploy --prod`
- [ ] Verify deployment logs show no errors
- [ ] Run smoke test checklist:
  - [ ] Frontend loads (HTTPS)
  - [ ] Wizard Step 1 → Step 3 works
  - [ ] Create incident API call succeeds (201 response)
  - [ ] GET /api/incidents returns list
  - [ ] PDF export initiates (no 500 errors)
- [ ] Check Lighthouse (should be ≥80 at this stage before optimization in 13-02)
- [ ] Verify database connections in Neon dashboard (should show active connections during load)

## Deliverables
- [x] Neon database created with backups enabled
- [x] Prisma migrations run on production database
- [x] `.env.example` updated with required variables
- [x] Vercel environment variables configured and secured
- [x] Frontend updated to use production API URL
- [x] Deployment logs verified; smoke tests passed
- [x] ROADMAP.md updated with completion status

## Notes
- **Critical:** Do not commit actual database URLs or secrets to git; use Vercel environment management
- **Monitoring:** After deployment, monitor Neon dashboard for connection pool saturation; if >90% utilization, increase pool max size
- **Rollback:** If deployment fails, use `vercel rollback` to revert to previous deployment
