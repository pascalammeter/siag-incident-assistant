# Phase 13-02 Plan: Performance Optimization & Lighthouse ≥90

## Overview
After deployment, audit Lighthouse score, optimize images, minimize JavaScript bundles, implement caching strategies, optimize database queries, and verify Core Web Vitals (LCP <2.5s, FID <100ms, CLS <0.1). Target score ≥90 across all Lighthouse categories.

## Success Criteria (MUST be TRUE)
1. ✅ Lighthouse performance score ≥90 (mobile and desktop)
2. ✅ Lighthouse accessibility score ≥90
3. ✅ Lighthouse best practices score ≥90
4. ✅ Lighthouse SEO score ≥90
5. ✅ Core Web Vitals: LCP <2.5s, FID <100ms, CLS <0.1 (verified on production)
6. ✅ API response times: average <200ms, p95 <500ms (verified with k6 load test)
7. ✅ Database queries optimized: no N+1 queries, indexes on all frequently queried columns

## Tasks

### 1. Lighthouse Audit & Initial Scoring (2 hours)
- [ ] Run Lighthouse audit on production URL using Chrome DevTools (mobile & desktop)
- [ ] Document baseline scores for: Performance, Accessibility, Best Practices, SEO
- [ ] Identify top performance issues (blocking resources, unused CSS/JS, images)
- [ ] Check Core Web Vitals in Google PageSpeed Insights or Chrome User Experience Report
- [ ] Generate report and share baseline metrics

### 2. Frontend Optimization (3 hours)

#### Images & Assets
- [ ] Audit all images for size/format; convert to WebP with JPEG fallback
- [ ] Implement `<Image>` component from `next/image` for automatic optimization
- [ ] Add lazy loading to below-the-fold images
- [ ] Optimize logos (SVG if possible; max 50 KB)
- [ ] Configure Vercel Image Optimization (free tier: 1000 requests/day)

#### JavaScript & CSS
- [ ] Run `npm run build` and check bundle size with `next/bundle-analyzer`
- [ ] Identify and tree-shake unused dependencies
- [ ] Code-split route-specific bundles (Wizard screens should lazy-load)
- [ ] Minimize Tailwind CSS bundle: verify `tailwind.config.ts` purges unused classes
- [ ] Remove unused fonts; use system font stack or load only required weights (e.g., 400, 600, 700)

#### Server-Side Rendering & Caching
- [ ] Enable ISR (Incremental Static Regeneration) for static pages (if applicable)
- [ ] Set appropriate Cache-Control headers for API responses
  - `/api/incidents` — max-age: 300 (5 min)
  - `/api/incidents/:id` — max-age: 600 (10 min)
  - Static assets — max-age: 31536000 (1 year)
- [ ] Enable compression (gzip, brotli) on Vercel (enabled by default)

#### Fonts & Typography
- [ ] Verify fonts are loaded via next/font (not @import); use font-display: swap
- [ ] Load only required font weights and styles
- [ ] Add `preload` hints for critical fonts in `<head>`

### 3. Backend Optimization (2.5 hours)

#### Database Queries
- [ ] Review all Prisma queries for N+1 patterns using Prisma Studio or logs
- [ ] Add `.include()` or `.select()` to prevent over-fetching
- [ ] Verify indexes exist on:
  - `incidents(created_at)` (for sorting/filtering)
  - `incidents(user_id)` (for list queries)
  - `incidents(incident_type)` (for filtering)
  - `audit_logs(incident_id)` (for incident detail page)
- [ ] Run EXPLAIN ANALYZE on top 5 slow queries; optimize if needed

#### API Response Optimization
- [ ] Implement response pagination: max 50 incidents per page (reduce payload size)
- [ ] Add `compression` middleware (gzip automatic on Vercel; optional for local)
- [ ] Set appropriate Content-Type and Content-Encoding headers
- [ ] Test API response times with k6 (use 12-02 load test):
  - GET /api/incidents — target <100ms
  - POST /api/incidents — target <200ms (includes DB write)
  - PDF export — target <2000ms (acceptable for heavy operation)

#### Function Optimization
- [ ] Review Vercel Function memory allocation; increase if p95 latency >500ms
- [ ] Check cold start times in Vercel deployment logs
- [ ] Implement connection pooling (already configured via Neon in 13-01)
- [ ] If using external services (e.g., PDF generation), implement caching or async processing

### 4. Production Monitoring Setup (1.5 hours)
- [ ] Install Vercel Analytics (free tier)
- [ ] Configure Web Vitals tracking in app layout
- [ ] Set up Vercel Monitoring dashboard to track:
  - Response time p50, p95, p99
  - Function invocation count
  - Error rate
  - Cold starts
- [ ] Create dashboard or alerts if metrics exceed thresholds

### 5. Re-audit & Verification (1.5 hours)
- [ ] Run Lighthouse audit again on production
- [ ] Verify all scores ≥90
- [ ] Check Core Web Vitals on real device (mobile) or use PageSpeed Insights
- [ ] Run k6 load test from 12-02 to verify API response times
- [ ] Document final scores and improvements vs. baseline

- [ ] Document optimization changes in DEVELOPMENT.md
