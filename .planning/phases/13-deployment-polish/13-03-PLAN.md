# Phase 13-03 Plan: Backwards Compatibility & Migration

## Overview
Ensure v1.1 deployment is fully backwards compatible with v1.0. All v1.0 URLs remain functional, wizard workflows unchanged, and localStorage incidents automatically migrate to the new API-backed system on first v1.1 load without user intervention.

## Success Criteria (MUST be TRUE)
1. ✅ All v1.0 URLs still accessible (no 404 redirects breaking bookmarks)
2. ✅ v1.0 wizard workflows function identically (Steps 0-6, same playbooks, same compliance logic)
3. ✅ localStorage incidents from v1.0 automatically migrate to API on first v1.1 load
4. ✅ No breaking changes to incident data model (all v1.0 fields preserved and backward compatible)
5. ✅ API gracefully degrades if backend unavailable (localStorage fallback works)
6. ✅ v1.0 & v1.1 can coexist on same domain without data collision

- [ ] Verify v1.1 deployment has identical route structure

  - [ ] `/` loads wizard

- [ ] Set up HTTP redirects for deprecated v1.0 URLs (if any) with 301 (permanent) status
- [ ] Document any URL changes in migration guide for users

### 2. Data Model & API Compatibility (2 hours)
- [ ] Review Prisma schema: verify all v1.0 fields preserved

- [ ] Verify incident object shape includes:
  - `id`, `title`, `description`, `incident_type`, `severity`
  - `playbook_steps` (array with done/notes fields)
  - `notifications` (compliance deadlines)
  - `created_at`, `updated_at`
- [ ] Ensure API errors use consistent format (e.g., `{ error, message, code }`)
- [ ] Test API response pagination doesn't break existing integrations
- [ ] Document API breaking changes (if any) in CHANGELOG.md with migration instructions

### 3. localStorage to API Migration (3 hours)

#### Migration Logic Implementation
- [ ] Create `migrationService.ts` utility:
  - Detect if user has localStorage incidents and no API incidents
  - Parse localStorage incidents (validate against v1.0 schema)
  - Call POST /api/incidents for each localStorage incident
  - Handle API errors gracefully (show toast, retry option)
- [ ] Add migration trigger in `useIncident()` hook:
  - On first `useIncident()` call, check localStorage for incidents
  - If found and API is reachable, run migration in background
  - Show notification: "Migrating your incidents from local storage... this may take a minute"
  - Update progress as incidents upload
  - Show completion message when done

#### Error Handling & Fallback
- [ ] If API is unreachable during migration:
  - Continue using localStorage (graceful degradation)
  - Show warning: "Backend is currently unavailable; using local storage. Your data will sync when available."
  - Periodically retry migration (every 5 minutes)
- [ ] If migration partially completes (some incidents uploaded, then API fails):
  - Store migration state in localStorage (last incident uploaded ID)
  - Resume from that point on retry
  - Don't double-upload incidents

#### User Communication
- [ ] Add migration notice in header/sidebar during first load:
  - "We've upgraded to v1.1! Your incidents from local storage are being migrated to secure cloud storage."
- [ ] Show progress bar or spinner while migration in flight
- [ ] Disable "Create New Incident" button during migration (prevent race conditions)
- [ ] On completion, show success toast: "Your incidents are now backed up in the cloud"
