## Interaction States

### Loading State (new -- wizard resume fetch in-flight)

| State | Visual | Duration |
|-------|--------|----------|
| Fetch in-flight | `LoadingSpinner size="lg"` centered in wizard container (`flex justify-center items-center py-12`). No wizard steps rendered. No progress bar rendered. | Until API response or timeout |
| Fetch success | Spinner removed. Wizard renders at Step 1 with hydrated data. No toast. | Instant transition |
| Fetch fail + localStorage hit | Spinner removed. Wizard renders at Step 1 with cached data. Warning toast shown. | Instant transition + 4s toast |
| Fetch fail + no cache | Spinner removed. Error toast shown. `router.push('/incidents')` after 1500ms delay to allow toast to be read. | 1500ms then redirect |
| 404 response | Spinner removed. Error toast shown ("Vorfall nicht gefunden oder geloescht"). `router.push('/incidents')` after 1500ms delay. | 1500ms then redirect |

### Hydration Guard (existing pattern, extended)

The `WizardProvider` currently returns `null` until `isHydrated` is `true` (localStorage read complete). This phase extends the guard:

1. If `incidentId` is NOT provided: existing behavior unchanged (localStorage hydration only)
2. If `incidentId` IS provided: `isHydrated` remains `false` until the API fetch completes (or falls back). The `LoadingSpinner` is shown during this period by the parent component (`WizardShell` or a wrapper).

### Step 0 Skip (resume flow)

When `incidentId` is provided and data is hydrated:
- `currentStep` is set to `1` (Einstieg), not `0` (Interstitial)
- The Interstitial splash screen is irrelevant for existing incidents (CONTEXT.md D-07)
- Progress bar starts at Step 1
